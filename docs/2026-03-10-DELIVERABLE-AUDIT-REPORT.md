## Critical Issues Found

### ISSUE 1: XP Cell Format Mismatch (CRITICAL)

**Severity:** 🔴 CRITICAL — Binary format incompatibility

**Python Implementation (xp_core.py):**
```python
out_buffer.write(struct.pack('<I', glyph))  # 4 bytes - unsigned int
out_buffer.write(bytes(fg))                 # 3 bytes
out_buffer.write(bytes(bg))                 # 3 bytes
# Total: 10 bytes per cell
```

**JavaScript Implementation (xp-file-reader.js):**
```javascript
const BYTES_PER_CELL = 7;  // glyph(1) + fg(3) + bg(3)
```

**JavaScript Implementation (xp-file-writer.js):**
```javascript
cellBytes.push(cell.glyph & 0xFF);         // 1 byte - uint8
cellBytes.push(cell.fg[0] & 0xFF);         // 3 bytes RGB
cellBytes.push(cell.bg[0] & 0xFF);         // 3 bytes RGB
// Total: 7 bytes per cell
```

**Impact:**
- ❌ Python-generated .xp files (10 bytes/cell) CANNOT be read by JavaScript (expects 7 bytes/cell)
- ❌ JavaScript-generated .xp files (7 bytes/cell) would FAIL when read by Python (expects 10 bytes/cell)
- ❌ Glyph values > 255 would be truncated in JavaScript (8-bit) vs preserved in Python (32-bit)

**Evidence:**
- Python doc: "Cell format: glyph_u32 | fg_r | fg_g | fg_b | bg_r | bg_g | bg_b (10 bytes)"
- JavaScript reader: Line 178-181: "Cell format: 7 bytes per cell"
- JavaScript writer: Line 35-42: "Push" 7 individual bytes

**What the game expects:**
- Real .xp files in `sprites/` were generated by Python (10 bytes/cell format) ✓ **VERIFIED**

**VERIFICATION (actual sprite file analysis):**
```
File: sprites/player-0100.xp
Decompressed size: 403,240 bytes
Header: 20 bytes + (4 layers × 12 bytes) = 68 bytes
Cell data: 403,240 - 68 = 403,172 bytes
Cells: 126 width × 80 height × 4 layers = 40,320 cells
Bytes per cell: 403,172 ÷ 40,320 = 10.00 ✓

Expected for 7 bytes/cell: 40,320 × 7 = 282,240 bytes ✗ (WRONG)
Expected for 10 bytes/cell: 40,320 × 10 = 403,200 bytes ✓ (CORRECT)
```

**Definitive proof:** Real game sprite files use **10-byte format**.
If JavaScript tries to load this file with 7-byte assumption, it will read past cell boundaries.

**Fix Required:**
- Update JavaScript reader to use `BYTES_PER_CELL = 10`
- Update JavaScript writer to write glyph as 4-byte uint32 (little-endian)
- Update documentation to reflect the correct 10-byte format

## Root Cause Analysis

**Why does the mismatch exist?**

1. **Python format** (xp_core.py) uses true REXPaint format (10 bytes/cell)
   - Generated for actual game use
   - Used throughout the pipeline

2. **JavaScript format** (xp-file-reader.js, xp-file-writer.js) uses simplified format (7 bytes/cell)
   - Appears to be a custom simplification
   - Incompatible with actual game .xp files
   - Tests only use synthetic data (7-byte format)

3. **No integration tests** between Python and JavaScript formats
   - Tests don't try to load real `sprites/*.xp` files
   - Tests don't verify cross-language compatibility
   - Unit tests pass in isolation but fail in real-world use
