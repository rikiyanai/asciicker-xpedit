## Conclusion

WASM crash is **strongly supported** as caused by a race condition in `override_mode=mounted`
XP injection, correlated with file size / injection duration.

**Key evidence chain:**
1. A/B matrix: pipeline (18KB) crashes 7/10 while native (4.2KB) crashes 0/10
2. A/A/B causal check: pipeline content at native dimensions (126x80) crashes 5/5 — **rejects dimensional mismatch as sole cause**
3. Layer structure fix (populate layers 0/1): no crash reduction — **rejects layer encoding as cause**
4. Hybrid test (native content + blank layer 1): 0 crashes — confirms native content is safe
5. Identical XP files produce stochastic results across runs — **confirms non-determinism / race condition**
6. Crash rate correlates with XP file size → injection duration → race window width

**The geometry contract improvement (commit b196f39) is valid for XP format correctness
but does not address the crash.** The root cause is that `override_mode=mounted` writes
XP bytes to Emscripten FS while WASM is already running, and larger files widen the
window for memory access conflicts.

**Caveat**: the native control baseline is crash-free but not fully valid — all 10
native runs are `invalid_run` with `classification=unknown` due to pos=[None,None,None].

**Next steps:**
1. Fix the injection race: either gate WASM startup until injection completes, or use
   a pre-boot injection mode that writes files before WASM initializes
2. Investigate pos=[None,None,None] classification regression (separate PR)
